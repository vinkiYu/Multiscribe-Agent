# 后置包大纲（P14–P24）

> 这些包是 MVP（P0–P13）之后的增量。此处只给**目标/范围/验收锚点/依赖**大纲。
> **临近执行时再细化为完整可执行包**（与 P0–P13 同格式）——避免 MVP 阶段的学习成果被提前固化导致返工。
> 执行顺序建议按依赖图，但可按优先级调整。

---

## P14 — 其余采集适配器

**目标**：补全 Follow API、GitHub Trending、AI 搜索三个适配器。

**范围**：
- `plugins/builtin/adapters/follow.py`（FollowApiAdapter：分页游标 + Cookie/Folo 头 + 详情二次拉取 + 限速）。
- `plugins/builtin/adapters/github_trending.py`（GitHubTrendingAdapter：httpx + 正则爬取 + UA 轮换反爬）。
- `plugins/builtin/adapters/ai_search.py`（AISearchAdapter：调 Agent/Workflow executor + extract_json 解析）。
- 各自 config_fields + 测试（fixture/respx mock）。
- OPML 导入端点（`POST /api/adapters/import-opml`，fast-xml-parser → feed 列表）。

**验收锚点**：三适配器各能抓取真实/fixture 数据 → UnifiedData；被 discovery 注册；AI 搜索能调 executor 返回结构化结果。

**依赖**：P6（RSS 适配器模式）。

## P15 — 其余发布端（GitHub / RSS / 微信公众号）

**目标**：补全原项目发布能力。

**范围**：
- `plugins/builtin/publishers/github.py`（PyGithub Content API：get-sha → createOrUpdate，422 幂等）。
- `plugins/builtin/publishers/rss.py`（feedgen 本地生成 XML）。
- `plugins/builtin/publishers/wechat/`（公众号 API 发文：access_token 管理 + 过期刷新锁、Markdown→HTML renderer、图片管线 PIL 压缩/视频 ffmpeg 转 GIF/双通道上传 CDN+素材库、并发 Semaphore 限流、指数退避、news/newspic）。
- 发布历史端点（`/api/history/*`）、`/api/publish/:id`。
- CommitRecord 持久化。

**验收锚点**：GitHub 发文件成功；RSS 生成合法 XML；公众号发文成功（含图片上传）；commit_history 记录。

**依赖**：P7/P8（publisher 模式）。

## P16 — 知识库 + 混合检索

**目标**：实现分层索引 + 向量召回 + FTS5 bm25 + RRF 融合 + 可选重排。

**范围**：
- `infra/embedding.py`（sentence-transformers 本地 / 可选 API；批量编码；sqlite-vec 存储）。
- `knowledge/document_processor.py`（PDF pypdf / DOCX python-docx / XLSX openpyxl / CSV / MD 解析 + 滑动窗口分块 + sha256 去重）。
- `knowledge/hierarchical_service.py`（root.json → categories → documents → chunks/*.md 三层索引）。
- `knowledge/retriever.py`（向量 top-k + FTS5 top-k → RRF 融合 → 可选 LLM 重排；FTS5 降级模式）。
- `knowledge/kb_service.py`（分类/文档/分块 CRUD + 查询 + move-to-memory）。
- `infra/db.py` 补 kb_* 表业务 + embeddings 表 + FTS5 触发器激活。
- API：`/api/kb/*`。

**验收锚点**：多格式文档上传→分块→建索引→混合检索召回相关分块；RRF 排序合理；FTS5 降级可用。

**依赖**：P2（DB），P3（LLM 重排可选）。

## P17 — 记忆系统

**目标**：分层记忆 + 混合检索 + 跨分类去重 + LLM 分类。

**范围**：
- `memory/memory_service.py`（分层 root.json → categories → entries/*.md；sha256 跨分类去重；LLM 分类；当日条目排除；合并）。
- 复用 P16 的混合 retriever。
- `infra/db.py` 补 agent_memories 表业务 + FTS5。
- API：`/api/memory/*`。
- 与 P4 HarnessContext 的 `inject_memory` 接通。

**验收锚点**：记忆去重；LLM 分类正确；混合检索召回；当日排除；merge 工作。

**依赖**：P16（混合检索复用）。

## P18 — MCP 客户端

**目标**：接入外部 MCP 服务器作为 Agent 工具来源。

**范围**：
- `agents/mcp_service.py`（`mcp` SDK：stdio/sse/streamable_http 三传输；schema 清洗 cleanMCPSchema；命名隔离 `safeId__safeTool`；客户端缓存；disconnect_all）。
- AgentExecutor 的 MCP 工具路由（本地工具优先，否则 MCP；MCP 结果兜底解析 `{content:[{text}]}`）。
- API：`/api/mcp-configs` CRUD（删除时清理 Agent 引用）。
- AgentDefinition.mcp_server_ids 接通。

**验收锚点**：配一个 MCP server（如 stdio filesystem）→ Agent 能调用其工具；命名隔离生效；reload 断连。

**依赖**：P4（AgentExecutor），P1（MCPConfig）。

## P19 — Skill 系统

**目标**：指令注入 + execute_command 联动 + Skill 商店同步。

**范围**：
- `agents/skill_service.py`（扫描 `./skills` `./data/skills`；解析 frontmatter + 正文；依赖检查 bins；build_skills_prompt）。
- `agents/skill_sync_service.py`（文件系统 → store diff 同步）。
- AgentExecutor：绑定 skill_id 时自动加 execute_command 工具。
- 内置 skills 移植（memory-read/write 等，按需）。
- API：`/api/skills/*`（scan/store search/github import/file CRUD）。

**验收锚点**：Skill 注入 prompt；execute_command 联动；文件系统同步增删改。

**依赖**：P4，P5（execute_command 工具）。

## P20 — 精简前端

**目标**：React + Vite + Tailwind，5 核心页面。

**范围**：
- `frontend/`：Login / Dashboard（状态+日志+手动触发）/ Agents&Workflows（配置+运行+SSE 预览）/ Knowledge（知识库+记忆+检索测试）/ Settings（AI/插件/调度/推送端配置）。
- 复用后端 SSE 事件流。
- FastAPI 静态托管 frontend/dist。

**验收锚点**：5 页可用；SSE 实时显示；配置可保存触发 reload。

**依赖**：P12（API 齐全），P16/P17（Knowledge 页）。

## P21 — 评估框架

**目标**：数据集 + LLM-as-judge + Benchmark 回归。

**范围**：
- `eval/dataset.py`（YAML/JSON 数据集管理：input + expected/参考）。
- `eval/evaluator.py`（跑 Agent/Workflow，按维度评分：LLM rubric + 规则匹配 + 相似度）。
- `eval/benchmark.py`（批量跑 + 聚合：平均分/通过率/regression 对比基线）。
- CLI：`multiscribe-agent eval --dataset xxx --agent yyy`。
- 预置每日摘要质量 eval 数据集。

**验收锚点**：CLI 跑数据集出报告；regression 检测（分数下降告警）。

**依赖**：P4（Agent），P11（摘要场景）。

## P22 — Interop 互操作层

**目标**：外部 AI 自助注册 → 审批 → 发现 → 执行。

**范围**：
- `services/interop.py`（API Key 生成 sk_ + sha256 hash；注册/审批/验证；能力发现；OpenAI Function 格式工具导出；配置脱敏 + 增量合并；统一执行网关）。
- API：`/api/ai/v1/{register,verify,discovery,context,tools,skills,settings,schedules,agents,workflows,execute}`。
- API Key 双轨认证（X-API-Key，仅 v1/*）。
- INTEROP_GUIDE.md（给 AI 读的操作手册，Jinja2 动态填充能力列表）。

**验收锚点**：外部 AI 走注册→审批→执行闭环；配置脱敏；增量合并不覆盖密钥。

**依赖**：P12（API/认证），P5（工具导出）。

## P23 — 完整可观测性（OTel）

**目标**：全链路 trace + metrics + Prometheus 端点。

**范围**：
- `observability/tracer.py` + `meter.py`（OTel 初始化）。
- Agent run/round/tool/workflow 全 span；token/latency/工具调用/推送成功率 metric（counter/histogram）。
- `/metrics`（Prometheus 格式）端点。
- structlog → OTel trace 关联（trace_id 注入日志）。

**验收锚点**：Jaeger/控制台见全链路 span；/metrics 可被 Prometheus 抓取。

**依赖**：P12（基础 structlog）。

---

## P24 — Loop Engineering 深化

**目标**：把 Loop 从单节点扩展为系统级「执行→评估→精炼→收敛」范式。

**范围**：
- 增强 Planner/Reflector（多维度 rubric、自修复建议生成）。
- Workflow 的 Loop 节点支持更丰富 exit_condition（复合规则、指标阈值）。
- 与 P21 评估框架联动：评估不达标自动触发精炼工作流。
- 沉淀 skill `loop-engineering-patterns`。

**验收锚点**：复杂任务能多轮迭代收敛；评估驱动自动精炼；有 benchmark 对比单轮 vs 多轮的质量提升。

**依赖**：P10（Loop 节点），P21（评估）。
