# 大纲：P29 — P2/P3 长期优化

> **阶段**：阶段五（架构债清理，长期）
> **目标**：修复评估报告 P2/P3 级缺陷，向企业级平台演进。
> **依赖**：P26/P27/P28 全部通过后，按业务需要排期。
> **状态**：⚪ 未开始（持续优化，不设硬通过节点）。

---

## 一、定位与原则

P29 不是单个任务包，而是**一组可独立排期**的优化项大纲。每项在执行前需单独生成详细任务包（沿用 P25.1/P26/P27/P28 格式）。

**优先级原则**：
- P2 项：影响用户体验或运维便利性，建议在 P1 全部修复后 6 个月内排期。
- P3 项：纯优化，不影响功能，按性能压测结果决定是否投入。

---

## 二、P2 项（次要缺陷）

### P2-1：配置版本控制

**现状**：`SystemSettings` 无 schema 版本字段；升级时可能出现兼容性问题。

**目标**：
- `SystemSettings` 新增 `config_version: str` 字段。
- 启动时检测版本，不兼容时拒绝启动 + 输出迁移指引。
- 提供 `multiscribe-agent config migrate --from 0.1 --to 0.2` CLI 子命令。

**预估**：2 天。
**依赖**：无。

---

### P2-2：工作流暂停/恢复机制

**现状**：工作流一旦启动必须跑完或超时；不支持人工审批节点。

**目标**：
- `WorkflowEngine` 支持 `pause(workflow_run_id)` / `resume(workflow_run_id)`。
- 新增 `type: "human_approval"` 节点类型，执行到该节点自动暂停，等待外部 `approve()` 调用。
- 暂停状态持久化（基于 P1-4 的 `workflow_iterations` 表扩展）。

**预估**：5 天。
**依赖**：P1-4（P26 已实现 iteration_store）。

---

### P2-3：数据库备份机制

**现状**：无自动备份；数据丢失风险。

**目标**：
- CLI 子命令 `multiscribe-agent db backup --output data/backups/{timestamp}.sqlite`。
- 封装 SQLite `.backup` 命令（在线备份，不阻塞写入）。
- 可选 cron 集成（保留最近 N 份）。

**预估**：1 天。
**依赖**：无。

---

### P2-4：插件热重载

**现状**：插件修改需重启服务。

**目标**：
- 用 `watchdog` 监听 `plugins/custom/` 目录文件变化。
- 变更触发 `ServiceContext.reload_plugins()`（保留旧实例直到新实例就绪）。
- 热重载事件通过 P1-3 的 EventBus 广播。

**预估**：3 天。
**依赖**：P1-3（P26 EventBus）、P1-9（P28 版本检查）。

---

### P2-5：密钥轮换机制

**现状**：JWT secret 与 API Key 无轮换；密钥泄露后无法平滑切换。

**目标**：
- 支持 `JWT_SECRET_CURRENT` + `JWT_SECRET_PREVIOUS` 双密钥并存。
- 登录签发用 current；验证接受 current 或 previous。
- 提供 `multiscribe-agent rotate-secret` CLI 子命令。

**预估**：2 天。
**依赖**：无。

---

### P2-6：插件契约测试

**现状**：无自动化契约验证；插件接口变更可能破坏兼容。

**目标**：
- `tests/contracts/` 新增契约测试套件。
- 自动发现所有 `plugins/builtin/` 与 `plugins/custom/` 下的插件。
- 验证：Adapter 实现 `fetch` + `transform`；Publisher 实现 `publish`；metadata 字段齐全。

**预估**：2 天。
**依赖**：P1-9（P28 版本字段）。

---

## 三、P3 项（优化建议）

### P3-1：Token 估算精度提升

**现状**：`context.py` 用 `(len+3)//4` 启发式，对中文严重低估（1 中文字 ≈ 1-2 token，但启发式按 4 字符 ≈ 1 token）。

**目标**：
- 接入 `tiktoken`（OpenAI 模型）或 provider 的 token counting API。
- 中文字符按 1.5x 权重估算（fallback when tiktoken 不可用）。
- 估算误差从当前 ~40% 降到 < 10%。

**预估**：3 天。
**依赖**：无。

---

### P3-2：流式工具调用

**现状**：工具调用结果一次性返回；长耗时工具（如 web 抓取）首字节延迟高。

**目标**：
- `Tool` 协议新增 `async def stream_invoke(args) -> AsyncIterator[ToolChunk]`。
- executor 把 chunk 流转发给上层 SSE。
- 兼容既有 `invoke()`（非流式工具自动包装为单 chunk）。

**预估**：5 天。
**依赖**：无。

---

### P3-3：数据库读副本

**现状**：单 SQLite 文件；读 QPS 受限。

**目标**：
- 评估 PostgreSQL 迁移（高并发写场景）。
- 或 SQLite Litestream 实时复制到只读副本（读扩展）。
- 提供 `DatabaseRouter` 抽象，读写分离透明。

**预估**：2 周（含压测）。
**依赖**：P1-7（P28 连接池）。

---

### P3-4：完整 Jaeger/Zipkin 集成

**现状**：P23 已实现 OTel tracer + Console/OTLP exporter；但无可视化 UI。

**目标**：
- 提供 `docker-compose.observability.yml`（Jaeger + Prometheus + Grafana）。
- 文档化 `OTEL_EXPORTER_OTLP_ENDPOINT` 配置。
- 预置 Grafana dashboard 模板（Agent run latency / token usage / publish success）。

**预估**：3 天。
**依赖**：P23（已完成）。

## 四、执行建议

### 短期（P26/P27/P28 完成后 1-2 个月）

优先级排序：
1. **P2-3 数据库备份**（1 天，低成本高收益，立即做）
2. **P2-5 密钥轮换**（2 天，安全基线）
3. **P2-6 插件契约测试**（2 天，防回归）
4. **P3-1 Token 估算精度**（3 天，直接影响 Agent 质量）

### 中期（3-6 个月）

5. **P2-1 配置版本控制**（2 天）
6. **P2-4 插件热重载**（3 天）
7. **P3-4 Jaeger/Grafana 集成**（3 天）

### 长期（6-12 个月，按业务需要）

8. **P2-2 工作流暂停/恢复**（5 天，需人工审批场景才做）

10. **P3-3 数据库读副本/PG 迁移**（2 周，需 500+ QPS 场景才做）
