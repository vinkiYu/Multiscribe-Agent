# P0.5 — MVP 默认配置绑定（前置修复包）

> **状态**：未开始 · **前置依赖**：P12 已通过 · **阻塞**：P13

## 目标

打通 MVP 端到端的最后 50 米：让 `.env` 中的单值凭据变量（`OPENAI_API_KEY` / `FEISHU_WEBHOOK` 等）自动映射到结构化 `ProviderConfig` / `PublisherConfig`，并由 `bootstrap` 启动时创建默认 curation `AgentDefinition`。修复后，`uv run python -m multiscribe_agent digest` 只需填 `.env` 即可跑通，无需手动 API 创建 AgentDefinition。

## 根因分析

```
.env: OPENAI_API_KEY=sk-xxx      ← 存在
     ↘
SystemSettings.ai_providers[default-openai].api_key  ← 未映射，始终为空
     ↘
create_provider() → ProviderError("no api key configured")
     ↘
P4/P11 LLM 调用全挂

.env: FEISHU_WEBHOOK=https://open.larks...   ← 存在
     ↘
SystemSettings.publishers[feishu_bot].config    ← 未映射
     ↘
PublisherConfig.enabled                        ← 始终 False（default_false）
     ↘
PublishingService fanout 跳过飞书端

.env: 无 curator agent
     ↘
DailyDigestConfig.curate_agent_id 无默认值
     ↘
CLI digest 无法构造默认任务包
```

## 详细任务

### T1. `config.py` — 环境变量 → Provider/Publisher 配置映射

#### 1a. Provider api_key 映射

在 `SystemSettings` 中利用 `pydantic_settings` 的 `field_env`（或 `ValidationInfo` 回调）机制，将 `.env` 中的 `OPENAI_API_KEY`、`ANTHROPIC_API_KEY`、`GOOGLE_API_KEY` 映射到对应 `ProviderConfig[id=default-openai].api_key` 等。

方案选择（见下「映射方案决策」）。

#### 1b. Publisher webhook 映射

将 `FEISHU_WEBHOOK` → `publishers[id=feishu_bot].config["webhook"]`；将 `FEISHU_SECRET` → `publishers[id=feishu_bot].config["secret"]`；将 `WECOM_WEBHOOK` → `publishers[id=wecom_bot].config["webhook"]`。

当对应 `.env` 变量非空时，publisher `enabled` 设为 `True`（覆盖 `_default_publishers()` 中的 `False` 默认值）。

#### 1c. MVP 默认 daily_digest 配置块

在 `SystemSettings` 中新增字段：

```python
default_curation_provider_id: str = "default-openai"
default_curation_model: str = "gpt-4o-mini"
default_curation_temperature: float = 0.3
default_digest_targets: list[str] = Field(default_factory=lambda: ["feishu_bot", "wecom_bot"])
default_digest_top_n: int = 5
default_digest_fetch_days: int = 2
default_digest_adapter_ids: list[str] = Field(default_factory=lambda: ["rss-adapter"])
```

`DEFAULT_CURATION_PROVIDER_ID` / `DEFAULT_CURATION_MODEL` 等新增字段可在 `.env.example` 中注释说明。

### T2. `bootstrap.py` — 默认 Curation AgentDefinition bootstrap

在 `ServiceContext.init()` 末尾（或 `run_daily_digest_task()` 中惰性创建），检查 `entities.get("agents", "default-curation-agent")` 是否存在；若不存在，用 `settings.default_curation_provider_id` 和 `settings.default_curation_model` 创建并持久化一个最小 `AgentDefinition`：

```python

        name="Default Curation Agent",
        description="MVP 默认精选 agent，由 bootstrap 自动创建",
        provider_id=self.settings.default_curation_provider_id,
        model=self.settings.default_curation_model,
        temperature=self.settings.default_curation_temperature,

            "输出 JSON 数组，每项包含 id/title/summary/score（0-10）。"
        ),
    )
    await self.entities.save("agents", "default-curation-agent", default_def.model_dump())
```

## 映射方案决策

两种实现路径，Codex 按包执行前先在白名单内二选一（不需要规划额外决策）：

### 方案 A：`model_validator` 后处理（推荐）

```python
class SystemSettings(BaseSettings):
    # ... 现有字段 ...
    openai_api_key: str = ""
    anthropic_api_key: str = ""
    feishu_webhook: str = ""
    feishu_secret: str = ""
    wecom_webhook: str = ""

    @model_validator(mode="after")
    def _bind_env_to_providers(self) -> SystemSettings:
        for p in self.ai_providers:
            if p.type == "openai" and self.openai_api_key:
                p.api_key = self.openai_api_key
            if p.type == "anthropic" and self.anthropic_api_key:
                p.api_key = self.anthropic_api_key
            # google...
        for p in self.publishers:
            if p.id == "feishu_bot":
                if self.feishu_webhook:
                    p.config["webhook"] = self.feishu_webhook
                    p.enabled = True
                if self.feishu_secret:
                    p.config["secret"] = self.feishu_secret
            if p.id == "wecom_bot":
                if self.wecom_webhook:
                    p.config["webhook"] = self.wecom_webhook
                    p.enabled = True
        return self
```

优点：符合 Pydantic v2 `model_validator` 语义；单向 `.env → 内存配置`，不破坏序列化；不需改 `ProviderConfig` / `PublisherConfig` 结构。

### 方案 B：`SettingsConfigDict.env_prefix` + 嵌套模型

将每个 provider/publisher 声明为带 `env_nested` 的子模型，并用 `MULTISCRIBE_PROVIDERS__DEFAULT_OPENAI__API_KEY` 格式。

缺点：需多级嵌套映射；`.env.example` 格式复杂（`PROVIDERS__ID__API_KEY`）；与现有 `_default_*` 工厂函数冲突。

**推荐方案 A**。
