# P30.1 Digest Curate 投影精简加固修复报告

**版本**：P30.1  
**修复提交**：`ba6e6e3 fix(digest): harden curate projection size`  
**报告日期**：2026-07-24  
**状态**：已实现、已验证、待远端推送

## 1. 修复结论

本次修复解决了 P30 上下文窗口优化中 curate 投影精简比例不足的问题。原实现虽然把 `description` 从完整正文截断到 500 字符，但在真实 RSS 短摘要和中等摘要场景中，仍然保留了大量相对无效的结构字段，因此 Prompt 体积相对原始 `UnifiedData` 只能减少约一半，无法满足“精简比例低于 30%”的阶段目标。

修复后，curate 阶段的 LLM 输入投影固定为三个字段：

```json
{
  "id": "source-id",
  "title": "资讯标题",
  "summary": "最多 150 个字符的描述片段"
}
```

URL、source 和 category 不再进入 curate Prompt。精选返回后，代码继续通过 `id -> UnifiedData` 回映原始记录，因此发布阶段仍能拿到原始 URL 和来源信息。overview/fanout 使用的 `_digest_item_dict` 没有修改。

## 2. 问题表现与根因

### 2.1 原始实现

修复前 `_curate_item_dict` 保留六个字段：

```python
{
    "id": item.id,
    "title": item.title,
    "summary": item.description[:500],
    "url": item.url,
    "source": item.source,
    "category": item.category,
}
```

这导致短摘要场景的 `summary` 截断几乎没有作用，真正被去掉的只有 `author`、`metadata`、`status` 等字段；而 `url/source/category` 对当前 curate Prompt 的打分和回写并非必需。

### 2.2 真实场景复现

用本阶段测试相同的 `UnifiedData` 构造方式，以 JSON 字符长度作为阶段验收口径，重算旧投影比例：

| 场景 | 旧投影比例 | 目标 |
| --- | ---: | ---: |
| 100 条 RSS 短摘要 | 50.9916% | `< 30%` |
| 20% 长 + 50% 中 + 30% 短混合分布 | 59.4522% | `< 30%` |

根因不是单纯的 `description` 上限过大，而是两个问题叠加：

1. 500 字符上限对 RSS 短摘要和中等摘要不起作用；
2. curate 阶段保留了回映后才能使用的 URL、source、category。

## 3. 修复方案

### 3.1 缩短摘要上限

将 `daily_digest.py` 中：

```python
_CURATE_SUMMARY_CHAR_LIMIT = 500
```

改为：

```python
_CURATE_SUMMARY_CHAR_LIMIT = 150
```

150 字符足以让模型判断资讯主题、相关性和重要性，同时可以显著限制长正文、README 和搜索结果进入上下文。

### 3.2 删除非必要字段

`_curate_item_dict` 现在只返回：

- `id`：用于把 LLM 结果映射回原始 `UnifiedData`；
- `title`：提供标题和主题上下文；
- `summary`：提供受限的内容片段。

精选阶段的既有回映逻辑保持不变：

```python
by_id = {item.id: item for item in items}
source = by_id.get(item_id)
url = source.url
source = source.source
```

因此该优化只改变模型输入，不改变下游发布数据契约。

### 3.3 明确不修改的范围

以下内容保持原样：

- `CURATE_PROMPT` 文本；
- `_digest_item_dict`；
- overview 节点输入；
- fanout 节点输入和 `CuratedDigest`；
- AgentExecutor、HarnessContext、TokenCounter；
- Provider、配置、数据库和插件实现。

这样可以把风险限定在 curate Prompt 投影，不引入跨阶段行为变化。

## 4. 修复后的实测结果

使用与旧实现相同的数据构造和 JSON 字符长度计算：

| 场景 | 旧实现 | 新实现 | 改善 |
| --- | ---: | ---: | ---: |
| RSS 短摘要 | 50.9916% | 27.4958% | 降低 23.4952 个百分点 |
| 混合分布 | 59.4522% | 25.2384% | 降低 34.2138 个百分点 |

两种场景均低于 30% 验收线。新实现还额外验证了：

- 投影字段集合严格等于 `{"id", "title", "summary"}`；
- 500 字符描述最终只保留 150 字符；
- 短描述不被补齐或改写；
- 100 条候选的整体 Prompt 比例低于 30%。

这里的比例是任务包规定的 JSON 字符长度比例，不是 tokenizer Token 比例。实际模型 Token 仍受 P30 的 `TokenCounter`、模型窗口、输出预留和安全余量控制；如果后续要把“低于 30%”改成 Token 口径，应单独增加 tokenizer 基准测试，不能直接复用字符比例结论。

## 6. 兼容性分析

### 对 LLM 输入的影响

模型看不到 URL、source、category，但仍看到稳定 ID、标题和内容片段。当前 Prompt 要求模型返回 ID，且代码只接受能回映到原始候选的 ID，因此不会因为投影字段减少而丢失发布链接。

### 对精选结果的影响

摘要从最多 500 字符变为最多 150 字符，模型能看到的长文上下文减少。该取舍是为了保证候选规模扩大时上下文预算稳定；完整正文此前也没有直接进入发布，而只是用于 curate 阶段判断。对于确实需要全文事实核验的场景，应另设检索或证据工具，不应把完整正文重新塞回所有候选 Prompt。

### 对 overview/fanout 的影响

没有影响。精选结果仍被转换成 `DigestItem`，其 URL/source 来自原始 `UnifiedData`；overview 和渠道渲染继续使用现有的 `_digest_item_dict` 和 `CuratedDigest`。

## 7. 风险与后续建议

1. **字段依赖风险**：如果未来精选排序要按来源、分类或域名加权，需要显式扩展 curate 投影契约和对应测试。
2. **摘要截断风险**：150 字符是字符数上限，中文、Emoji、HTML 片段的 Token 成本不完全相同；建议后续在 P30.2 用真实 tokenizer 建立 Token 比例门禁。
3. **事实完整性风险**：短摘要可能截断关键事实。可考虑让适配器先生成结构化关键句，或在模型要求证据时通过工具按 ID 按需读取正文。
4. **比例口径风险**：当前 `< 30%` 是 JSON 字符长度验收，并非 Provider 实际计费 Token；不要将两者混用。
5. **Prompt 契约风险**：本包没有新增对 `score_reason`、score 范围和摘要语言/长度的运行时校验，这些属于独立的精选质量加固问题。
